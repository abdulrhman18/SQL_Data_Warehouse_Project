# Naming Conventions

This document defines the naming standards used across the Data Warehouse environment.  
Applying consistent naming conventions ensures better **readability, maintainability, and scalability** while allowing teams to work efficiently with the data warehouse structure.

# General Guidelines

The following principles apply to all database objects within the data warehouse:

- **Naming Style**  
  Use `snake_case` formatting (lowercase letters with underscores) for all object names.

- **Language Standard**  
  All names must be written in **English** to ensure consistency and collaboration across teams.

- **Clarity and Meaning**  
  Object names should clearly describe their **business purpose and meaning**.

- **Avoid SQL Reserved Words**  
  SQL keywords such as `SELECT`, `TABLE`, or `GROUP` must not be used as object names.

- **Consistency**  
  Maintain consistent naming patterns across schemas, tables, columns, and procedures.

# Table Naming Standards

Table naming conventions follow the **data warehouse layered architecture**.  
Each layer serves a different purpose in the data pipeline.

---

# Bronze Layer

The **Bronze Layer** stores raw data exactly as it is received from source systems without transformation.

### Naming Pattern


<sourcesystem>_<entity>


### Components

- **sourcesystem** → The origin system of the data (e.g., `crm`, `erp`)
- **entity** → The original table name from the source system

### Example


crm_customer_info


Meaning: Raw customer information extracted from the CRM system.

### Rules

- Table names should **match the original source tables**.
- No renaming or structural transformation should occur in this layer.
- The Bronze layer acts as the **raw ingestion layer**.

---

# Silver Layer

The **Silver Layer** contains cleaned, standardized, and transformed data derived from the Bronze layer.

### Naming Pattern


<sourcesystem>_<entity>


### Components

- **sourcesystem** → Original system where the data originated
- **entity** → Dataset representing the cleaned version of the source table

### Example


crm_customer_info


Meaning: Cleaned and standardized CRM customer data.

### Rules

- Data is **validated and standardized**.
- Missing or inconsistent values are handled.
- The structure remains closely aligned with the source system.

---

# Gold Layer

The **Gold Layer** represents the **business-ready analytical layer** used for reporting, dashboards, and business intelligence.

### Naming Pattern


<category>_<entity>


### Components

- **category** → The analytical structure type
- **entity** → Business-oriented table name

### Examples


dim_customers
fact_sales
report_sales_monthly


These tables are optimized for **analytics, reporting, and dashboarding tools**.

---

# Category Prefix Reference

| Prefix | Description | Example |
|------|------|------|
| `dim_` | Dimension tables containing descriptive attributes | `dim_customer`, `dim_product` |
| `fact_` | Fact tables storing measurable business events | `fact_sales` |
| `report_` | Aggregated reporting tables | `report_sales_monthly` |

# Column Naming Standards

Column names must follow consistent and descriptive patterns to maintain clarity across the data warehouse.

---

# Surrogate Keys

Dimension tables use **surrogate keys** as primary identifiers.

### Naming Pattern


<table_name>_key


### Example


customer_key


Meaning: Surrogate primary key in the `dim_customers` table.

### Rules

- Surrogate keys should be used in **dimension tables**.
- They should be **system-generated** rather than derived from source systems.

---

# Technical Columns

Technical columns are metadata fields generated by the data warehouse system.

### Naming Pattern


dwh_<column_name>


### Example


dwh_load_date


Meaning: Timestamp indicating when the record was loaded into the data warehouse.

### Common Technical Columns

- `dwh_load_date`
- `dwh_update_date`
- `dwh_source_system`

These columns help with **data lineage, auditing, and pipeline monitoring**.

---

# Stored Procedure Naming

Stored procedures responsible for **loading data between layers** must follow a standardized naming pattern.

### Naming Pattern


load_<layer>


### Examples


load_bronze
load_silver
load_gold


### Meaning

- **load_bronze** → Loads raw data from source systems into the Bronze layer.
- **load_silver** → Cleans and transforms Bronze data into the Silver layer.
- **load_gold** → Builds analytical models and aggregates for reporting.

# Summary

Applying consistent naming conventions across the data warehouse ensures:

- Clear and structured data architecture
- Easier collaboration between data engineers and analysts
- Improved maintainability and scalability
- Faster onboarding for new team members

These conventions create a **standardized foundation for building reliable and scalable data warehouse solutions**.
